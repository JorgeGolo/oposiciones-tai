Aprovisionamiento y Configuración de la Base de Datos

Guía paso a paso para desplegar Azure SQL Database de forma 100% gratuita y conectarla de forma segura sin exponer credenciales.

1. Crear Azure SQL Database con la Oferta Gratuita

Azure incluye un asistente simplificado que aplica la capa gratuita de forma automática. Para configurarla sin errores:

  1. Ve al Portal de Azure, busca SQL databases en la barra superior y haz clic en Create.
  2. Verificación de la Oferta: Asegúrate de que aparece el banner gris arriba del todo que indica: "Oferta gratuita aplicada. Obtendrá 100 000 segundos de núcleo virtual, 32 GB de datos...". En el lateral derecho, el resumen de costos debe marcar Gratis.
  3. En Detalles del proyecto, selecciona tu Suscripción y tu Grupo de Recursos existente (rg-oposicion-tai).
  4. Baja en la pantalla hasta la sección Detalles de la base de datos (rellena lo siguiente):
    • Nombre de la base de datos: Introduce db-oposicion-tai.
    • Servidor: Haz clic en Crear nuevo. Ponle un nombre único (ej. sql-server-oposicion-jorge), elige la región (West Europe) y selecciona autenticación de tipo SQL Authentication. Elige un usuario administrador (ej. dbadmin) y una contraseña fuerte. (¡Guarda bien estos datos!).
  5. Haz clic en Siguiente: Redes (Networking) en la parte inferior.
  6. CRUCIAL: En la pestaña de Redes, busca la opción "Permitir que los servicios y recursos de Azure accedan a este servidor" (Allow Azure services and resources to access this server) y márcala como Sí. Si no lo haces, tu API de .NET no podrá leer las tablas desde el App Service.
  7. Haz clic en el botón azul Revisar y crear (Review + create) del principio y confirma la creación.

Es necesario crear un servidor de la base de datos.

2. Seguridad: Gestión del Connection String (Sin Datos Sensibles)

Para cumplir con las directivas de seguridad y no subir contraseñas a GitHub, dividimos la configuración en dos entornos:

En Local (Desarrollo)

Guardamos la cadena de conexión en el archivo appsettings.Development.json del backend, el cual debe estar incluido en el archivo .gitignore: 

{
  "ConnectionStrings": {
    "DefaultConnection": "Server=tcp:tu-servidor.database.windows.net,1433;Initial Catalog=db-oposicion-tai;Persist Security Info=False;User ID=tu_usuario;Password=TU_CONTRASEÑA_REAL;MultipleActiveResultSets=True;Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;"
  }
}

En Producción (Azure App Service)

En el archivo de producción appsettings.json que se sube al repositorio, dejamos la cadena vacía o con un token genérico: 

{
  "ConnectionStrings": {
    "DefaultConnection": "ProductionConnectionPlaceholder"
  }
}

La contraseña real se inyecta directamente desde el panel de Azure de forma encriptada:

  • Ve a tu App Service (el backend) en el portal de Azure.
  • En el menú lateral, entra en Settings → Configuration (o Environment variables según la interfaz actual).
  • Añade una nueva Connection String con el nombre DefaultConnection, pega la cadena con la contraseña real y selecciona el tipo SQLAzure. Azure priorizará este valor oculto sobre el del archivo de código.

3. Configuración de Entity Framework Core (Code-First)

Pasos fundamentales para preparar el backend en .NET Core, mapear los modelos de la Oposición y generar las tablas en Azure SQL.

Paso 1: Instalar los paquetes necesarios en el Backend

Abre la terminal de VS Code dentro de la carpeta de tu proyecto de .NET (Web API) y ejecuta estos tres comandos para instalar las herramientas de Entity Framework: 

# El motor principal de Entity Framework
dotnet add package Microsoft.EntityFrameworkCore.SqlServer

# Las herramientas para poder hacer migraciones desde la consola
dotnet add package Microsoft.EntityFrameworkCore.Design

# (Opcional pero recomendado) Para que tu app lea bien las configuraciones
dotnet add package Microsoft.EntityFrameworkCore.Tools

Paso 2: Crear tu Modelo (La tabla del CRUD)

Vamos a crear un modelo sencillo para empezar a probar el CRUD (por ejemplo, una tabla de Temas para tu oposición). Crea una carpeta llamada Models (si no la tienes) y dentro un archivo Tema.cs: 

namespace TuProyecto.Models
{
    public class Tema
    {
        public int Id { get; set; }
        public string Titulo { get; set; } = string.Empty;
        public string Bloque { get; set; } = string.Empty; // Ej: "Tecnología Básica"
        public bool Completado { get; set; } = false;
    }
}

Paso 3: Crear el DataContext

El DataContext es el puente entre tus modelos de C# y la base de datos de Azure SQL. Crea una carpeta Data y dentro el archivo ApplicationDbContext.cs: 

using Microsoft.EntityFrameworkCore;
using TuProyecto.Models;

namespace TuProyecto.Data
{
    public class ApplicationDbContext : DbContext
    {
        public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options) : base(options)
        {
        }

        // Esto le dice a EF que cree una tabla llamada "Temas" basada en el modelo anterior
        public DbSet<Tema> Temas { get; set; }
    }
}

Paso 4: Registrar el Contexto en Program.cs

Abre tu archivo Program.cs para decirle a tu API que use esa cadena de conexión que configuramos antes: 

using Microsoft.EntityFrameworkCore;
using TuProyecto.Data;

var builder = WebApplication.CreateBuilder(args);

// Configurar la conexión a SQL Server / Azure SQL
builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));

// ... resto de tu configuración (CORS, Controladores, etc.)

Paso 5: Lanzar las Migraciones a Azure

Con el código base guardado, abre la terminal en la raíz del backend y ejecuta los comandos de Entity Framework para impactar la nube:

  1. Crear la migración (prepara el plano de la tabla en código): 
    dotnet ef migrations add InitialCreate
    * Si la consola no reconoce "dotnet ef", ejecuta primero: dotnet tool install --global dotnet-ef
  2. Actualizar la Base de Datos (crea la estructura física real en Azure SQL): 
    dotnet ef database update

En cuanto termine el segundo comando, tu base de datos en Azure tendrá físicamente la tabla lista para recibir y persistir datos desde el frontend.

EXTRA

Configuración de Redes y Cortafuegos en Azure (Paso Obligatorio)

Por defecto, Azure bloquea cualquier intento de conexión externa. Configura estos dos interruptores en la pestaña de Redes del servidor (o en el menú lateral Seguridad → Redes):

  1. Acceso a la red pública: Asegúrate de cambiar el selector principal a "Redes seleccionadas" (Selected networks).
    * ADVERTENCIA: Elige esta opción en lugar de "Deshabilitar". Si lo dejas deshabilitado, Azure ignorará cualquier regla inferior y rechazará las conexiones de desarrollo local arrojando un error de Host desconocido.
  2. Regla de Cortafuegos Local (Firewall): Haz clic en el botón "Agregar la dirección IPv4 de cliente actual". Esto registrará la IP pública de tu casa (ej: 79.116.245.209) para habilitar que tu terminal local pueda lanzar las migraciones sin recibir el error Client IP address is not allowed. Recuerda darle a Guardar.

En Local (Desarrollo) - Formato de Contraseña Especial

Guardamos la cadena de conexión en el archivo appsettings.Development.json. Regla de oro: Si tu contraseña contiene caracteres especiales como # o ;, debes envolverla obligatoriamente entre comillas simples (') para que el analizador de .NET no confunda los caracteres con propiedades adicionales del string. 

{
  "ConnectionStrings": {
    "DefaultConnection": "Server=tcp:sql-server-oposicion-jorge.database.windows.net,1433;Initial Catalog=db-oposicion-tai;Persist Security Info=False;User ID=dbadmin;Password='Tu#Contraseña;Especial';MultipleActiveResultSets=True;Encrypt=True;TrustServerCertificate=False;Connection Timeout=30;"
  }
}

* REVISIÓN: El host debe terminar estrictamente en .windows.net y especificar el puerto de SQL Server (,1433) para evitar el error de Win32Exception 11001 (Host desconocido).

Paso 4: Registrar el Contexto y Habilitar Resiliencia en Program.cs

Para evitar fallos transitorios causados por la latencia o por el retraso de reactivación de la base de datos Serverless (capa gratuita que se duerme si no se usa), configura la estrategia de reintentos automáticos (EnableRetryOnFailure): 

using Microsoft.EntityFrameworkCore;
using TuProyecto.Data;

var builder = WebApplication.CreateBuilder(args);

builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("DefaultConnection"),
        sqlServerOptionsAction: sqlOptions =>
        {
            // Si Azure tarda en responder o despertar, .NET lo reintentará automáticamente
            sqlOptions.EnableRetryOnFailure(
                maxRetryCount: 6,
                maxRetryDelay: TimeSpan.FromSeconds(10),
                errorNumbersToAdd: null);
        }));

Migraciones automáticas al arrancar

Para que las tablas se creen automáticamente en Azure SQL cada vez que se despliega una nueva migración, añade este bloque en Program.cs justo después de var app = builder.Build();: 

using (var scope = app.Services.CreateScope())
{
    var db = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
    db.Database.Migrate();
}

Este bloque es idempotente: comprueba en la tabla interna __EFMigrationsHistory qué migraciones ya se han aplicado y solo ejecuta las pendientes. Si no hay ninguna nueva, no hace nada.

Importante: entornos de base de datos

El comando dotnet ef database update aplica las migraciones contra la base de datos que apunte tu cadena de conexión local (la de appsettings.Development.json). Es decir, crea las tablas en tu SQL Server local o en el SQL Azure si tu máquina tiene acceso directo, pero no tiene ningún efecto sobre la base de datos de producción en Azure durante el despliegue.

Por eso es imprescindible el bloque db.Database.Migrate() en Program.cs: es el único mecanismo que se ejecuta desde dentro del App Service de Azure, con la cadena de conexión de producción, y que por tanto puede crear o actualizar las tablas en Azure SQL automáticamente cada vez que se despliega una nueva versión.

Resumen práctico:

  • dotnet ef database update → aplica migraciones en local, lo ejecutas tú manualmente.
  • db.Database.Migrate() → aplica migraciones en producción, lo ejecuta la app sola al arrancar.
