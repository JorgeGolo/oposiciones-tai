Arquitectura de la Interfaz Frontend

Puntos clave sobre cómo se ha construido y estructurado la interfaz de usuario de este panel de estudio.

1. Angular & Angular Material 3

El proyecto utiliza Angular 18+ con componentes de tipo standalone y el sistema de diseño Material 3. Hemos seleccionado la paleta predefinida Azure/Blue durante la instalación, la cual inyecta los estilos globales e iconos de forma automática.

2. Sistema de Estilos Híbrido

Para evitar un diseño plano y mejorar el contraste visual, combinamos el potencial de ambos mundos:

  • Tokens Nativos: Mapeamos los colores directamente del inspector usando variables CSS globales como var(--mat-sys-primary) o var(--mat-sys-surface-container).
  • Clases Propias: Toda la estructura del Layout, la barra lateral (.sidebar) y la superior (.topbar) se controlan con clases SCSS personalizadas y encapsuladas.

3. Navegación SPA (Sidebar + Router)

El menú lateral utiliza <mat-nav-list> integrado con el enrutador nativo de Angular. Al usar routerLink en lugar de enlaces tradicionales, el contenido se intercambia al instante dentro del contenedor principal gracias a la directiva: 

<main class="content">
  <router-outlet></router-outlet>
</main>

La barra de rutas (app.routes.ts) gestiona la carga diferida (Lazy Loading) y la redirección automática al inicio: 

{ path: '', redirectTo: '/estructura', pathMatch: 'full' }

4. Modularidad con Angular CLI

Cada sección del panel se gestiona como un componente independiente y limpio generado directamente desde la terminal mediante la interfaz de comandos de Angular: 

ng generate component paginas/nombre-componente

CRUD de la tabla Temas

La estructura ideal de archivos (Arquitectura)

Dentro de tu carpeta temas/, la estructura limpia debería quedar así:

  • temas.service.ts: El cerebro. Contiene los métodos HTTP (GET, POST, PUT, DELETE) para pegarle a tu API de .NET.
  • tema-list/: El componente principal. Muestra la tabla o lista con todos los temas (Read) y tiene los botones de "Eliminar" (Delete).
  • tema-form/: El componente formulario. Sirve tanto para crear como para editar (Create y Update). Reutilizar el mismo formulario para ambas acciones te ahorra la mitad del código.

Flujo de trabajo en Angular

[ tema-list ] ──(Click en Editar)──> [ tema-form ]
      │                                    │
(Llama al Servicio)                (Llama al Servicio)
      │                                    │
      ▼                                    ▼
 [ temas.service ] <──────HTTP──────> [ API .NET Core ]
  1. El Servicio (temas.service.ts): Centraliza las llamadas a Azure. Tendrá funciones como getTemas(), createTema(tema), updateTema(id, tema) y deleteTema(id).
  2. La Lista (tema-list): Al arrancar, le pide los temas al servicio y los pinta en un bucle (*ngFor o arroba for). Si el usuario pulsa "Eliminar", llama al servicio y refresca la lista. Si pulsa "Nuevo" o "Editar", Angular Router lo lleva al formulario.
  3. El Formulario (tema-form):
    • Si entra por /temas/nuevo, se muestra vacío. Al enviar, hace un POST.
    • Si entra por /temas/editar/5, pide los datos del tema ID 5 al servicio, rellena los inputs, y al enviar hace un PUT.

¿Qué comandos lanzar en la terminal?

Para generar esta estructura limpia de una sola vez, colócate en la carpeta de tu frontend de Angular y ejecuta: 

# 1. Crear el servicio que conectará con .NET
ng generate service temas/services/temas

# 2. Crear el componente para listar y borrar
ng generate component temas/components/tema-list

# 3. Crear el componente para crear y editar
ng generate component temas/components/tema-form
